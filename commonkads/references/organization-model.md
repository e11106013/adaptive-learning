# 組織模型 (Organization Model) 詳細指引

CommonKADS 組織模型用於診斷組織脈絡中的知識問題，評估開發知識系統的商業價值。

---

## OM-1：組織情境描述

**目的**：建立整體組織背景，理解 KBS 開發的環境。

### 填寫項目

| 項目 | 說明 | 範例 |
|------|------|------|
| 組織名稱 | 分析對象的組織或部門 | 客服中心、承保部門 |
| 使命/目標 | 組織的核心業務目標 | 提供 24/7 客戶技術支援 |
| 規模與結構 | 人員規模、部門層級 | 50人、3層管理 |
| 業務流程 | 主要工作流程列表 | 故障診斷、報告撰寫 |
| 外部環境 | 法規、競爭、客戶需求 | GDPR 合規、客戶 SLA |
| 現有 IT 系統 | 目前使用的系統 | CRM、ERP、知識庫 |

### 填寫範本

```
組織名稱：___________
使命：___________
主要業務流程：
  1. ___________
  2. ___________
規模：___ 人，分 ___ 個部門
現有系統：___________
相關法規限制：___________
```

## OM-2：問題與機會描述

**目的**：識別組織中值得用 KBS 解決的知識問題。

### 問題識別清單

詢問以下問題來發現知識瓶頸：

- 哪些決策嚴重依賴少數專家？
- 哪些任務因為知識難以傳授導致訓練時間過長？
- 哪些流程因為知識分散導致結果不一致？
- 哪些錯誤是因為遺忘或疏忽造成的？
- 哪些知識因為人員離職而面臨流失風險？

### 填寫項目

| 項目 | 說明 |
|------|------|
| 問題描述 | 具體描述知識問題 |
| 問題來源 | 知識稀缺？不一致？難以傳遞？ |
| 影響範圍 | 影響哪些流程、多少人、多少成本 |
| 目前因應方式 | 組織現在如何處理這個問題 |
| 改善機會 | KBS 可以帶來什麼改善 |

### OM-2 工作表範本

```
問題識別碼：OM-2-[編號]
問題描述：
  ___________________________________________

問題類型（勾選）：
  □ 知識稀缺（只有少數人會）
  □ 知識不一致（不同人做法不同）
  □ 知識隱性（難以文件化）
  □ 知識流失風險（關鍵人員即將離職）
  □ 知識傳遞緩慢（訓練耗時）

業務影響：
  財務影響：___________
  品質影響：___________
  時間影響：___________

目前因應方式：___________
預期改善程度：___________
```

## OM-3：處理變體描述

**目的**：描述知識任務目前的執行方式，識別最佳實踐與變異。

### 分析框架

描述同一任務的「變體」（Variant）——不同人員、不同情境下如何處理：

| 變體 | 執行者 | 方法 | 品質 | 效率 |
|------|--------|------|------|------|
| V1（最佳） | 資深工程師 | 系統化診斷 | 高 | 快 |
| V2（平均） | 一般工程師 | 試誤法 | 中 | 慢 |
| V3（問題） | 新進人員 | 查手冊 | 低 | 非常慢 |

### 關鍵問題

- V1（最佳實踐）的知識在哪裡？可以形式化嗎？
- V2、V3 與 V1 的差距是什麼原因造成的？
- KBS 應該支援 V1 的推理過程還是取代 V2/V3？

## OM-4：資產描述（知識資產清單）

**目的**：識別組織中的知識資產，評估其可用性與存取性。

### 知識資產類型

| 類型 | 說明 | 範例 |
|------|------|------|
| 文件化知識 | 已記錄的知識 | 操作手冊、SOP、FAQ |
| 個人化知識 | 存在專家腦中的知識 | 老師傅的診斷直覺 |
| 嵌入式知識 | 存在系統/流程中的知識 | 程式邏輯、工作流程 |
| 文化性知識 | 組織慣例與默契 | 例外處理的不成文規定 |

### OM-4 工作表範本

```
資產識別碼：OM-4-[編號]
知識資產名稱：___________
類型：□文件化 □個人化 □嵌入式 □文化性
擁有者/位置：___________
存取難易度：□易 □中 □難
品質評估：□高 □中 □低
對 KBS 的相關性：□核心 □支援 □邊緣
取得授權需求：___________
```

## OM-5：知識系統行動計畫

**目的**：根據 OM-1～OM-4 的分析，提出具體的 KBS 開發建議。

### 行動計畫結構

1. **建議優先開發的知識問題**（來自 OM-2）
2. **目標解決方案描述**：KBS 的功能範疇
3. **預期效益量化**：成本節省、品質提升、時間縮短
4. **實施路線圖**：分階段計畫
5. **風險與假設**：前提條件與潛在風險

### OM-5 填寫範本

```
建議 KBS 名稱：___________
解決的核心問題（引用 OM-2 編號）：___________

功能描述：
  輸入：___________
  處理：___________
  輸出：___________

預期效益：
  效率提升：___________（如：縮短診斷時間 40%）
  品質提升：___________（如：錯誤率降低 30%）
  知識保存：___________（如：保存 3 位退休專家的知識）

實施優先序：□立即 □短期(3個月) □中期(6個月) □長期(1年+)
```

## OM-6：可行性評估

**目的**：從三個維度評估 KBS 開發的可行性，決定是否繼續。

### 三維可行性矩陣

| 維度 | 評估項目 | 評分（1-5） | 說明 |
|------|----------|-------------|------|
| **技術可行性** | 知識可形式化程度 | | |
| | 知識獲取難易度 | | |
| | 現有技術能力 | | |
| **組織可行性** | 管理層支持度 | | |
| | 用戶接受意願 | | |
| | 組織變革能力 | | |
| **財務可行性** | 開發成本估算 | | |
| | 預期 ROI | | |
| | 預算可得性 | | |

### 決策規則

- 三個維度均 ≥ 3：**建議繼續**，進入任務模型分析
- 任一維度 ≤ 2：**需要解決障礙**後再繼續
- 技術可行性 = 1：**不適合** KBS，考慮其他解決方案

## 組織模型分析完成清單

完成 OM 後應能回答：

- [ ] 組織中有哪些值得解決的知識問題？（OM-2）
- [ ] 最佳實踐知識存在哪裡？（OM-3、OM-4）
- [ ] 建議開發什麼樣的 KBS？（OM-5）
- [ ] 開發是否可行？（OM-6）
- [ ] 下一步是哪個任務的 TM 分析？（連接到任務模型）
