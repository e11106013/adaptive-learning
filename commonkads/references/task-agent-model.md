# 任務模型 & 代理模型 詳細指引

---

## 任務模型 (Task Model, TM)

**目的**：詳細描述 KBS 要支援的特定任務，識別其知識需求。

任務模型是組織模型（OM-5 行動計畫）與知識模型（KM）之間的橋樑。

---

## TM-1：任務描述

**一個 TM-1 對應一個需要 KBS 支援的任務。**

### 填寫項目

| 項目 | 說明 | 範例 |
|------|------|------|
| 任務名稱 | 簡短明確的任務名稱 | 設備故障診斷 |
| 任務目標 | 任務要達成什麼結果 | 找出故障原因並提出修復建議 |
| 任務類型 | 分類（見下方） | 診斷型 |
| 觸發條件 | 什麼情況下執行此任務 | 設備出現異常警報時 |
| 輸入 | 任務開始所需的資料/知識 | 錯誤碼、症狀描述、設備型號 |
| 輸出 | 任務產生的結果 | 故障原因、修復步驟、零件清單 |
| 控制 | 指導任務執行的規則/方針 | 安全規範、SOP 手冊 |
| 資源 | 執行任務需要的資源 | 診斷工具、知識庫 |
| 品質標準 | 如何評估任務執行品質 | 診斷準確率 > 90%，時間 < 30分鐘 |
| 執行頻率 | 每天/每週執行幾次 | 每天約 20 次 |
| 重要性 | 對組織目標的重要程度 | 高（直接影響客戶 SLA） |

### CommonKADS 任務類型分類

| 任務類型 | 描述 | 範例 |
|----------|------|------|
| **診斷** (Diagnosis) | 從症狀找出原因 | 設備故障診斷、疾病診斷 |
| **分類** (Classification) | 將案例分到預定類別 | 信用風險評級、客訴分類 |
| **設計** (Design) | 建立滿足需求的配置 | 產品配置、系統設計 |
| **規劃** (Planning) | 制定達成目標的行動序列 | 生產排程、專案規劃 |
| **監控** (Monitoring) | 持續觀察並在異常時警報 | 製程監控、財務監控 |
| **預測** (Prediction) | 預測系統未來狀態 | 設備壽命預測、需求預測 |
| **指導** (Instructing) | 提供逐步執行指引 | 故障排除程序、培訓輔助 |

### TM-1 填寫範本

```
任務識別碼：TM-1-[編號]
任務名稱：___________
任務類型：___________
觸發條件：___________

輸入：
  - [輸入1]：[說明]
  - [輸入2]：[說明]

輸出：
  - [輸出1]：[說明]
  - [輸出2]：[說明]

控制（規則/限制）：
  - ___________

資源：
  - ___________

品質標準：
  - 準確性：___________
  - 時間：___________
  - 完整性：___________

執行頻率：___________
重要性：□高 □中 □低
```

## TM-2：知識項目描述

**目的**：將任務分解為子任務，識別每個子任務的知識需求。

### 任務分解

先用 **GTM（Generic Task Model）** 或任務分解樹將 TM-1 的任務細化：

```
[診斷任務]
├── 1. 症狀收集 (Symptom Collection)
│   └── 知識需求：症狀識別標準
├── 2. 假說生成 (Hypothesis Generation)
│   └── 知識需求：症狀→故障對應規則
├── 3. 假說驗證 (Hypothesis Verification)
│   └── 知識需求：測試方法、判斷標準
└── 4. 解決方案生成 (Solution Generation)
    └── 知識需求：修復程序、零件規格
```

### TM-2 填寫範本（每個子任務一份）

```
子任務識別碼：TM-2-[編號]
子任務名稱：___________
所屬任務（TM-1 編號）：___________

知識需求：
  類型：□規則性 □案例性 □概念性 □程序性
  描述：___________
  來源：___________（引用 OM-4 資產編號）
  
知識可用性評估：
  □ 已文件化，可直接使用
  □ 存在專家腦中，需要知識獲取
  □ 尚未存在，需要建立
  
知識獲取方法：
  □ 文件分析  □ 專家訪談  □ 案例收集  □ 問卷調查
```

## 代理模型 (Agent Model, AM)

**目的**：描述執行（或將被 KBS 支援/取代）的代理人——可以是人員或系統。

---

## AM-1：代理描述

### 代理類型

| 代理類型 | 說明 |
|----------|------|
| 人員代理 | 執行任務的人員（如：工程師、審核員） |
| 系統代理 | 執行任務的軟體系統 |
| KBS代理 | 擬開發的知識系統本身 |

### AM-1 填寫項目

| 項目 | 說明 |
|------|------|
| 代理名稱 | 角色名稱（非個人姓名） |
| 代理類型 | 人員/系統/KBS |
| 執行任務 | 此代理負責哪些 TM-1 任務 |
| 能力 | 擁有哪些知識與技能 |
| 限制 | 知識缺口、時間/精力限制 |
| 授權範圍 | 被允許做哪些決策 |
| 互動對象 | 與哪些其他代理互動 |
| 責任 | 需對哪些結果負責 |

### AM-1 填寫範本

```
代理識別碼：AM-1-[編號]
代理名稱：___________
代理類型：□人員 □系統 □KBS

執行的任務（引用 TM-1 編號）：___________

能力描述：
  知識技能：___________
  工具使用：___________
  經驗程度：□新手 □中級 □專家

限制：
  知識缺口：___________
  時間限制：___________
  資源限制：___________

授權範圍：
  可以決定：___________
  需要升級：___________
  
互動對象（引用其他 AM-1 編號）：___________

KBS 介入方式：
  □ 完全自動化（KBS 取代此代理）
  □ 決策支援（KBS 輔助此代理）
  □ 知識查詢（此代理主動查詢 KBS）
```
