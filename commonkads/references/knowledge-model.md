# 知識模型 & 推理模型 詳細指引

---

## 知識模型 (Knowledge Model, KM)

CommonKADS 最核心的模型。知識模型將 TM-2 識別的知識需求，轉化為
可實作的形式化知識結構。

### 知識模型三層架構

```
┌─────────────────────────────┐
│     任務知識 (Task Knowledge)  │  「要做什麼」的問題求解方法
├─────────────────────────────┤
│   推理知識 (Inference Knowledge) │  「如何推理」的推理步驟
├──────────────┬──────────────┤
│ 領域知識     │  推理使用的  │  「知道什麼」的概念與關係
│ (Domain K.)  │  領域知識    │
└──────────────┴──────────────┘
```

## 領域知識建模

**目的**：建立問題領域的概念本體（Ontology）。

### 領域本體組成要素

| 要素 | 說明 | 範例（設備診斷） |
|------|------|-----------------|
| **概念**（Concept） | 領域中的重要實體類型 | 設備、故障、症狀、零件 |
| **屬性**（Attribute） | 概念的特性 | 設備.型號、故障.嚴重程度 |
| **關係**（Relation） | 概念間的連結 | 症狀 causes 故障 |
| **規則**（Rule） | 業務邏輯 | IF 症狀=過熱 AND 位置=馬達 THEN 故障=軸承磨損 |
| **案例**（Case） | 歷史實例 | [案例#1234] 設備A, 症狀X → 故障Y |

### 知識類型分類

| 知識類型 | 說明 | 表示方式 |
|----------|------|----------|
| **事實知識** | 靜態的概念定義 | 本體、資料庫 |
| **規則知識** | 條件-行動規則 | IF-THEN 規則 |
| **案例知識** | 歷史經驗 | 案例庫 |
| **程序知識** | 執行步驟 | 流程圖、演算法 |
| **後設知識** | 關於知識的知識 | 規則優先序、可信度 |

### 領域本體建模範本

```
概念定義：
  概念名稱：___________
  定義：___________
  屬性：
    - [屬性名]：[類型]（[值域]）
    - [屬性名]：[類型]（[值域]）
  
關係定義：
  關係名稱：___________
  來源概念：___________
  目標概念：___________
  關係類型：□is-a □part-of □causes □indicates □requires

規則定義（每條規則）：
  規則識別碼：KM-R-[編號]
  條件部分（IF）：
    - [條件1]
    - [條件2]
  結論部分（THEN）：
    - [結論]
  可信度：___ %（若適用）
  來源：___________（專家/文件）
```

## 推理知識建模

**目的**：建立「推理結構圖」（Inference Structure），描述推理步驟與知識流向。

### 推理結構的組成

| 元素 | 符號 | 說明 |
|------|------|------|
| **知識來源** (KS) | 橢圓形 | 執行一個推理步驟的功能 |
| **知識角色** (KR) | 矩形 | 推理過程中知識的狀態/分類 |
| **箭頭** | → | 知識流向 |

### 診斷任務推理結構範例

```
[可觀察值]          KS1：抽象化        [系統狀態描述]
(Observables) ──────────────────→ (System State)
                                        │
                                   KS2：假說生成
                                        │
                                        ↓
                                  [故障假說]
                                 (Hypotheses)
                                        │
              [測試結果]          KS3：假說驗證
              (Test Results) ──────────────────→ [確認故障]
                                              (Confirmed Fault)
                                                    │
                                             KS4：解決方案選擇
                                                    │
                                                    ↓
                                            [修復方案]
                                           (Repair Plan)
```

### 常用元類別（Metaclasses）

CommonKADS 提供標準化的知識角色分類：

| 元類別 | 說明 | 使用場景 |
|--------|------|----------|
| Observable | 可觀測的輸入資料 | 症狀、量測值 |
| Hypothesis | 待驗證的可能解釋 | 可能故障原因 |
| Finding | 已確認的事實 | 確認的故障 |
| Norm | 正常標準/規格 | 正常運作範圍 |
| Discrepancy | 與標準的偏差 | 異常指標 |
| Complaint | 用戶報告的問題 | 客訴內容 |
| Goal | 要達成的目標狀態 | 系統目標 |
| Plan | 行動序列 | 解決步驟 |

### 知識來源（KS）建模範本

```
KS 識別碼：KM-KS-[編號]
KS 名稱：___________
功能描述：___________

輸入知識角色：
  - [角色名]（元類別：___________）

輸出知識角色：
  - [角色名]（元類別：___________）

使用的領域知識：
  - [引用規則/概念/案例]

推理機制：
  □ 規則演繹   □ 類比推理（案例）  □ 統計推斷
  □ 限制滿足   □ 最佳化           □ 其他：___
```

## 任務知識建模

**目的**：描述高層次的問題求解策略（Task Methods）。

### 通用任務方法（Generic Task Methods）

| 任務類型 | 常用方法 |
|----------|----------|
| 診斷 | Heuristic Classification、Systematic Diagnosis |
| 分類 | Hierarchical Classification、Parametric Design |
| 設計 | Propose-Critique-Modify、Skeletal Design |
| 規劃 | Hierarchical Planning、Case-Based Planning |

### 任務方法描述範本

```
任務方法識別碼：KM-TM-[編號]
方法名稱：___________

  3. [步驟3描述]

  步驟1 → KM-KS-___
  步驟2 → KM-KS-___

終止條件：___________
失敗處理：___________
```

## 推理模型 (Reasoning Model, RM)

**目的**：將知識模型的推理結構（Inference Structure）與任務方法整合，
完整描述從輸入到輸出的完整推理過程。

推理模型 = 任務方法 + 推理結構 + 領域知識對應

### 完整推理圖繪製步驟

1. 確定任務的輸入和輸出（來自 TM-1）
2. 從任務方法確定推理步驟的順序
3. 為每個步驟建立 KS 節點
4. 定義每個 KS 的輸入/輸出知識角色
5. 指定每個知識角色的元類別
6. 連接所有元素形成完整推理圖

### 推理模型驗證清單

- [ ] 每個 KS 的輸入都有明確來源
- [ ] 每個 KS 的輸出都有明確去向
- [ ] 任務的最終輸出對應到 TM-1 的輸出
- [ ] 每個 KS 使用的領域知識已在 KM 中定義
- [ ] 知識角色的元類別選擇合理
- [ ] 推理流程能覆蓋所有業務情境
